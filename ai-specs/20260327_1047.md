# Startup persistence mode wiring (CLI + config -> App)

## Request

Wire the new persistence inputs into startup mode resolution so App can decide between `Local` and `Repo` persistence with precedence:

1. CLI flag set -> `Repo`
2. Else config `persistence = "repo"` -> `Repo`
3. Else -> `Local`

Required touchpoints were `src/config/mod.rs`, `src/app.rs`, and `src/main.rs`, with tests covering precedence and no regression of existing local-default behavior.

## Response / Implementation

- Added `PersistenceMode` enum (`Local`, `Repo`) in `src/config/mod.rs`.
- Added `resolve_persistence_mode(cli_repo_flag, config)` in `src/config/mod.rs` to enforce precedence (`CLI -> config -> default`).
- Extended `App` in `src/app.rs` with `persistence_mode: PersistenceMode` and threaded it through startup construction (`App::new` and `App::build`).
- Updated startup wiring in `src/main.rs` to resolve mode once from CLI + config, then pass resolved mode into `App::new`.
- Updated App session-loading startup paths to use a centralized mode-aware session load helper (`load_session_for_context`) so persistence mode participates in startup session resolution logic.

## Tests added/updated (logic coverage)

- Config precedence tests in `src/config/mod.rs`:
  - CLI forces `Repo` over config
  - Config `repo` resolves to `Repo` when CLI absent
  - Config `local` resolves to `Local`
  - Default with no CLI/config resolves to `Local`
- App test in `src/app.rs` validating build preserves resolved persistence mode on App state.
- CLI parse test coverage in `src/theme/mod.rs` kept aligned with existing `--in-repo` flag behavior.

## Notes for future work

- Startup now has a single resolved persistence-mode source of truth carried on `App`, which makes follow-up storage routing changes (true repo-local storage implementation vs local XDG storage) straightforward to implement without changing CLI/config precedence again.
